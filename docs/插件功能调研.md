## 当前上下文判断

本工程已经具备按 C 编号或自定义方式录入、订单导入、数量精度、库存状态、两级分类、搜索筛选、详情编辑、重复合并、EDA 模型匹配、常用库复制、原理图放置和 JSON 备份等主流程。

库存条目仍以当前数量快照为主，但 schema-v9 已增加逐条最低库存、收藏、数据手册、结构化仓位、“BOM 出库批次与撤销流水”、项目需求快照、采购记录和用户确认的替代料链接。BOM 流水只覆盖显式整批出库与撤销；普通编辑、订单入库和合并尚不是完整交易审计，不能把该窗口描述成所有出入库流水。

路线图 P0 已实现：桌面端固定 JSON 路径自动备份、失败隔离、严格恢复校验和恢复前快照保护均已接入。首次设置无需输入，插件直接在系统“文档”目录下创建专用文件夹和固定名称 JSON，试写成功后启用；后续库存写入更新同一份“最新快照”，不会重复弹窗。由于 `SYS_Storage` 不提供宿主级原子 CAS，跨运行实例和跨设备写入仍不具备强一致性保证。

截至本文创建时，工作区存在用户尚未提交的改动。下一轮开发必须先执行 `git status --short` 并阅读相关 diff，不得覆盖或回退这些改动。

## F1：当前图纸库存校验与缺料导出

### 目标

在不修改库存的前提下，读取当前原理图或 PCB 的器件需求，与个人库存进行一次性比较，帮助用户在采购或装配前发现缺件。

第一版应命名为“当前设计库存校验”或“库存需求校验”，不要描述为完整的“BOM 管理”。它不保存项目 BOM，不负责采购下单，也不自动出库。

### 第一版范围

1. 从当前 SCH/PCB 读取器件，跳过明确设置为“不加入 BOM”的元件。
2. 优先按规范化 C 编号聚合；同一 C 编号合并数量和位号。
3. 支持用户输入板数，默认值为 `1`，总需求量为 `单板数量 × 板数`。
4. 与当前库存快照匹配并输出以下状态：
   - `充足`：准确库存大于或等于需求量。
   - `不足`：准确库存小于需求量，并显示缺口。
   - `库存缺失`：没有匹配的库存记录。
   - `需复核`：库存是估算数量，不能给出确定结论。
   - `需盘点`：库存数量未知。
   - `身份未匹配`：图纸器件没有 C 编号；保留在报告中，不能静默丢弃。
5. 报告显示 C 编号、名称、厂家型号、封装、位号、单板用量、板数、总需求、当前库存、缺口和状态。
6. 支持筛选状态和导出缺料 CSV；导出内容必须包含未匹配和无法精确判断的条目。
7. 读取宿主失败时显示明确原因，并提供选择外部 BOM 文件的降级入口。

### 明确不做

- 不改变任何库存数量或状态。
- 不持久化项目、PCB 或 BOM 快照。
- 不自动选择替代料。
- 不把估算数量或未知数量当作 `0`。
- 不因 C 编号缺失而按名称静默匹配并自动扣减。

### 推荐模块边界

```text
src/features/design-stock-check/
  domain/       需求聚合、数量精度和缺口规则
  application/  执行一次校验、生成导出模型
  ports/        当前设计读取和报告导出契约

src/platform/jlceda-v3/
  eda/          SCH/PCB/BOM BETA API 适配与能力检测
  presentation/ 总览内报告模态和文件导入降级
```

领域层只接收规范化的器件需求，不引用全局 `eda` 对象或官方 SDK 类型。

### 验收重点

- 同一 C 编号的多个位号能正确聚合。
- 板数倍率不会重复乘算。
- 准确、估算、未知、已用完和缺失库存分别得到正确状态。
- 无 C 编号、重复 C 编号、空图纸和“不加入 BOM”行为明确。
- 宿主 API 不可用时不崩溃，并能改用文件导入。
- 校验和导出均不增加库存 revision。
- Web 与桌面端分别验证 SCH 和 PCB。

## F2：最低库存与补货筛选

### 目标

允许用户为常用器件设置最低库存阈值，在总览中识别需要补货的准确库存条目。

### 建议规则

- 条目阈值使用可选字段，例如 `minimumQuantity?: number`；未设置时不预警。
- 是否需要全局默认阈值留待用户决策，第一版优先使用单条目显式设置，避免所有旧数据突然出现预警。
- 只有准确数量参与确定性的“低库存”判断。
- 估算数量低于阈值时显示“可能低于阈值”，未知数量显示“需盘点”，不要混入准确的低库存计数。
- 已用完条目始终可以进入补货视图，但仍保留其数量精度。
- 总览增加低库存过滤和补货 CSV 导出；导出不等于采购订单。
- 字段进入库存文档时必须通过新 schema 迁移、归一化和序列化测试，不能只保存在 IFrame `localStorage`。

## F3：从图纸选中元件定位库存

### 目标

用户在原理图或 PCB 中选中一个元件后，可以直接打开现有库存总览，定位并高亮对应库存记录和仓位。

### 建议规则

- 优先读取选中元件的 `SupplierId`，其次使用厂家型号提供候选，不按模糊名称直接确认身份。
- 找到唯一库存记录时，在同一总览长会话中应用搜索并高亮，不新建第二套库存列表。
- 找到多个候选时让用户选择；没有记录时只提供“按当前器件新增”的入口，不自动写库存。
- 多选元件时第一版可以显示选择列表，不能无提示只处理第一个。
- 选中读取属于宿主能力，必须通过平台适配器做能力检测和失败降级。

## F4：EDA 模型候选搜索与人工确认

### 目标

当精确 C 编号未命中、条目没有 C 编号或 EDA 模型失效时，允许用户搜索多个候选模型并明确选择。

### 建议规则

- 复用现有 `EdaDeviceCatalog.search(query, limit)`，查询词可来自 C 编号、厂家型号或用户输入。
- 候选列表至少展示 C 编号、名称、厂家型号、封装、符号和封装可用性。
- 自动评分只用于排序，不能代替用户确认。
- 第一版不接入 AI 服务、不保存 API Key，也不上传库存或 BOM。
- 用户确认后继续使用现有 revision 校验和模型引用清洗规则。
- 查询失败、无结果、取消和结果过期均不关闭库存总览。

## F5：嘉立创包装二维码入库

### 目标

识别包装标签二维码中的 C 编号、型号和数量，并复用现有的 C 编号查询、预览和重复合并流程。

### 第一版建议

- 优先支持扫码枪键盘输入或粘贴二维码原文；摄像头扫描作为可选增强，以降低 Web/桌面权限差异。
- 兼容已调研格式中的 `pc`、`pm`、`qty` 字段，但必须保留原始输入预览和人工确认，因为包装格式可能变化。
- C 编号通过现有规范化规则校验，数量必须是有上限的正整数。
- 扫码只生成导入草稿，不直接增加库存；最终写入复用现有重复项预览和合并策略。
- 不引入 LEYE 本地服务、私有 URL Scheme、远程 WebSocket 或共享服务器。

## F6：通用 BOM 列映射与版本差异

### 目标

为非固定格式 BOM 提供工作表选择、自动列映射、人工修正和预览，并支持两份 BOM 的需求差异比较。

### 当前规则与输入边界

- 继续使用本工程已有的 PapaParse 和 SheetJS，不引入外部插件的解析器代码。
- 订单导入一次最多选择 100 个文件，每个文件不超过 10 MiB；工作簿最多读取 32 个工作表、每表 128 列和 10,000 行数据。
- BOM 库存校验或出库一次只处理 1 个文件，BOM 版本差异必须选择 2 个文件；每个文件不超过 10 MiB，工作簿最多读取 32 个工作表、每表 128 列和 10,000 行数据。
- 从竞品吸收表头别名、编码提示、重复目标列警告、工作表选择和映射保存交互。
- 列映射由 F1、F6 和 F7 复用；映射预览本身不写库存，只有 F7 经二次确认后才执行原子出库。
- BOM 版本比较按规范化 C 编号和位号输出新增、删除、数量增加、数量减少和身份变化。
- 没有 C 编号的行必须明确显示匹配依据，不允许以名称模糊比较后宣称精确差异。
- 订单导入与通用 BOM 导入保持不同语义；订单号和订单文件 SHA-256 防重复不能直接套用为 BOM 版本身份。

## F7：BOM 出库批次与撤销流水

### 数据边界

库存条目继续保存当前数量快照；显式 BOM 批量出库另以文档级批次与流水记录来源、扣减和撤销，避免逐条修改后留下半批成功。

### 当前实现规则

- 流水包含稳定 ID、条目 ID、操作类型、数量变化、变更前后数量、时间、来源和操作人可见备注。
- BOM 出库包含来源文件或需求快照指纹，重复执行时能够识别并阻止二次扣减。
- 一批出库要么全部成功，要么在 revision 冲突或校验失败时全部不写。
- 支持撤销，并记录撤销与原流水的关联，不删除历史记录。
- 估算或未知库存需要先由用户确认实际基数，不能直接做精确扣减。
- 出库前显示缺失、库存不足和无法判断项目；默认不允许产生负库存。
- 自动备份在库存主写入成功后触发，备份失败不回滚主操作，但必须持续告警。

### 仍然禁止

- 绑定到鼠标或调用放置 API 后自动扣库存。
- 只按项目名、文件名或显示名称判断重复出库。
- 逐行写入导致半批成功。

## F8：项目/PCB 需求快照、采购和成本

当前实现可保存项目/文档需求快照、确认重同步差异、聚合板数需求，并独立维护采购记录和成本。

- 快照使用项目 UUID、PCB UUID 和来源时间，不只保存项目名称。
- 保存单板用量与板数，不把计算后的总量当作唯一事实。
- 重新同步必须显示差异并由用户确认，不能静默覆盖历史快照。
- 采购记录与库存流水分开建模；采购数量不等于当前库存。
- 单价、平台、链接、订单号、日期和采购批次属于采购记录，不能塞进库存条目备注；采购日期按 `YYYY-MM-DD` 纯日历日期保存，读取旧时间戳时会归一化为该格式。
- 采购清单只根据需求缺口生成建议，不自动下单。
- 当前不会识别“同一物理板”的 SCH 与 PCB 是否重复；若同时保留两份对应快照，需求会分别累计。完成去重或关联设计前，用户必须只保留其中一份，或删除重复快照。
- 当前只捕获活动 SCH/PCB，不自动遍历、切换或同步整工程文档；整工程切换与标签页恢复不属于 `0.5.0` 手测范围。

## F9：库存元数据与用户确认替代料

- 收藏：为高频器件提供置顶或独立过滤，不改变分类归属。
- 列显隐：保存用户的总览列偏好，系统关键列不可全部隐藏。
- 数据手册：保存或打开已存在的 Datasheet URL，不引入 PDF OCR 和 AI 问答。
- 结构化仓位：在确有标准器件柜时，将自由文本位置扩展为柜、盒、行、列；迁移时必须保留原位置文本。
- 替代料候选：只保存用户确认的等价关系，按足量、有库存、无库存排序，不能根据名称或封装自动宣称兼容。

## 明确暂不移植

- 放置元件后自动扣库存。官方放置 API 只能证明元件已绑定到鼠标，不能证明用户完成放置。
- 竞品的存储层、整文档覆盖、无 revision 写入和直接数量扣减。
- 手写 CSV 解析器。现有 PapaParse 和 SheetJS 更适合作为统一基础。
- Web Serial、BLE、固定器件盒协议、本地后台服务和远程扫码服务器。
- 完整交互式 PCB/BOM 渲染器。
- AI/OCR、外部模型 API Key 和库存数据上传。
- 抓取立创商城 HTML、在插件中嵌入共享密钥或把 `LIB_Device.search` 宣传为实时商城接口。
- 根据名称、封装或关键词自动写入分类、替代关系或用户已维护的商品字段。
- 完整项目文件树、BOM 生命周期或自动整工程同步体系。

## 必须保持的工程约束

1. `src/features` 不得引用全局 `eda` 或官方 SDK 类型，所有宿主 API 放在 `src/platform/jlceda-v3`。
2. IFrame 只提交业务意图，不直接拥有库存仓储写权限；写入继续由应用服务执行 revision 校验。
3. 新字段必须有 schema 迁移、旧数据读取、序列化、备份恢复和不支持未来 schema 的测试。
4. 商城商品身份与 EDA 模型引用继续独立；无模型库存仍可参与数量和缺料分析。
5. 数量的准确、估算和未知语义必须贯穿阈值、缺料、导出和出库，不能被统一转换成数字。
6. BETA API 必须有能力检测、诊断和降级路径；发布前分别验证嘉立创 EDA V3 Web 与桌面端。
7. 现有订单导入解析、文件上限、防重复和合并逻辑优先复用，外部插件代码只能补充明确缺口。
8. 新窗口继续遵循现有长会话和同一模态栈规则，不为每项功能重新创建独立窗口体系。
9. 日志不得记录完整 BOM、库存内容、本地路径或二维码原文；只记录步骤、数量、能力状态和错误类别。
10. 实现 F1 或其他 BOM 相关功能时，要同步修改 `inventory-overview-boundary.md`，明确“一次性只读库存校验”与“持久化 BOM 管理”的区别。

## 0.5.0 实现后实机回归清单

以下项目用于代码实现、自动化验证和构建完成后的宿主实机回归。回归时应记录环境、步骤、预期、实际结果和诊断日志中的错误类别。

### 基础录入与身份

- 按有效 C 编号添加，验证商品身份和 EDA 模型分别保存。
- C 编号无 EDA 模型、查询失败或用户取消商城核对时，确认仍可按既有规则保存。
- 自定义元器件添加，验证数量精度、状态、分类、位置和备注。
- 重复条目预览、合并和 revision 冲突。

### 订单导入

- `.xls`、`.xlsx`、CSV 和 JSON 的正常导入。
- 一次选择 100 个以内文件、单文件 10 MiB 以内的正常边界，以及超过文件数量、容量、32 个工作表、128 列或 10,000 行数据时的拒绝行为。
- 多工作表、表头偏移、空行、非法数量和不支持格式。
- 相同订单号或相同 SHA-256 文件的重复阻止。
- `add`、`replace`、`skip` 策略，以及导入中取消和失败后的库存一致性。

### 库存总览

- 搜索、清空搜索、分类、库存状态、模型状态、排序、分页和滚动位置恢复。
- 常驻总览的隐藏、恢复、原生最小化和显式关闭。
- 分类新增、重命名、排序、删除、批量移动及删除非空分类。
- 详情、完整编辑、删除确认、批量删除和 revision 冲突。
- 模型重新匹配的成功、无结果、失败、重试、取消和过期响应。
- 简体中文与英文、窄窗口和大数据量下的文本与控件布局。

### EDA 联动与备份

- 复制模型到个人库后的回读验证。
- 原理图放置只绑定到鼠标，不自动扣减库存。
- 手动 JSON 备份导出由系统保存对话框处理目的地、取消和覆盖确认。
- 桌面端自动备份首次设置时不出现路径输入或覆盖确认，直接创建默认目录和固定“最新快照”；同时验证后续更新不弹窗、失败持续告警但不回滚已提交的主库存操作。
- 恢复严格校验、恢复前快照和主恢复失败后的恢复点还原。
- Web 与桌面端分别验证 BETA API 能力和降级行为。

### BOM 出库与项目规划

- BOM 库存校验/出库只接受 1 个文件，版本差异只接受 2 个文件；逐项验证 10 MiB、32 个工作表、128 列和 10,000 行数据边界。
- 验证 BOM 出库预览、语义重复阻止、新生产确认、revision 冲突、原子扣减、活动批次条目保护和整批撤销。
- 新建采购记录时只保存 `YYYY-MM-DD`，读取旧时间戳时按兼容规则归一化为同一格式。
- 对同一物理板分别保存 SCH 与 PCB 快照，确认当前会重复累计；实机回归期间按现有边界删除一份，避免把重复需求带入采购建议。

### 测试结果收口

- 实机回归发现缺陷后，应先固定最小复现并补自动化回归，再继续发布收口。
- 将实机结论同步到对应边界文档、CHANGELOG 或专门测试记录，不只保留在聊天中。
- 无法确认是否属于环境差异的问题，应保留最小复现、客户端版本、Web/桌面环境和诊断日志类别。

## 已采用的实现决策

1. F1 同时支持 SCH 与 PCB；任何宿主读取失败都保留外部 BOM 文件降级入口。
2. F2 只使用逐条最低库存，不引入会让旧数据突然告警的全局默认值；估算和未知数量保留各自语义。
3. F5 只做扫码枪/粘贴和人工预览，不申请摄像头权限。
4. F7 先建立文档级原子批次、流水、语义重复识别和整批撤销，再开放写库存入口。相同语义需求的下一次生产必须二次确认，重新导出文件不能绕过去重。
5. F8 长期保存项目/文档 UUID、单板需求、板数和采购记录；重同步先展示差异，不静默覆盖。
6. F9 替代关系使用文档级无序条目对，由用户显式确认；关联条目删除或作为会消失的合并源时阻止操作，未关联来源合入已关联目标时保留目标及其关系，不做名称或封装兼容推断。

## 仍需用户验收或后续决策

- 分别在 Web 与桌面端确认 SCH/PCB 读取、选中元件、模型搜索、文件选择、IFrame 和路径写入能力。
- 使用真实 BOM、不同编码文本、真实包装码和大库存验证列映射、窗口布局、分页、焦点及导出内容。
- 确认项目快照与采购记录的实际使用频率，再决定是否增加归档、批量编辑或多币种换算；当前不做汇率换算。
- 确认是否需要 BOM 列映射预设和摄像头扫码；第一版没有实现这两项。
- 完成跨设备验证前，不把 `SYS_Storage` 描述为可靠云同步；跨运行实例 CAS 仍是已知宿主限制。

以上项目不影响自动化验证结论，但会影响是否可以发布为“已完成宿主兼容验证”。

## 0.5.0 实机回归顺序

1. 先备份现有真实库存，并记录 EDA 客户端版本、Web/桌面环境和插件版本 `0.5.1`。
2. 先测 P0 默认文件夹与固定最新快照自动创建、恢复、旧 schema 迁移、最低库存/元数据编辑和普通库存 CRUD，确认基础写入稳定。
3. 再测 F1/F3/F4 的 SCH 与 PCB 读取、外部 BOM 降级、精确定位高亮和模型候选确认。
4. 使用测试库存验证 F7 预览、重复阻止、新生产确认、并发冲突、整批撤销，以及活动批次条目删除/合并保护。
5. 最后测试 F8/F9 长会话、项目重同步差异、采购日期归一化、同一物理板 SCH/PCB 快照重复累计边界、采购导出、替代关系管理和大数据分页。
6. 把实机结果、客户端版本和诊断错误类别写回本文或专门联调记录；任何缺陷先补最小复现与自动化回归。

完成标准不是“界面可以打开”，而是业务规则、旧数据迁移、失败降级、revision 一致性、文档和实机验证均已收口。
