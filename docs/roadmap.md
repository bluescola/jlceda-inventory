# 后续功能路线

本文记录已经明确的可靠性路线及其状态。P0 已在 0.5.0 实现，仍需在嘉立创 EDA Web/桌面宿主和多设备环境完成的项目继续保留为 P1/P2；实际状态以 README、CHANGELOG、代码和测试记录为准。

## 设计原则

- EDA 用户配置继续作为运行时主存储，保持现有 Web/桌面兼容性和开箱即用体验。
- 本地文件是可恢复的备份副本，不作为可独立修改的第二主数据源。
- 主存储与备份严格按顺序写入：先确认 EDA 主存储成功，再生成本地备份。
- 自动备份失败不得回滚已经成功的库存操作，但必须向用户显示可持续追踪的失败状态。
- 所有恢复操作必须先验证、再预览、后确认，不直接覆盖当前库存。

## P0：指定路径自动备份与恢复（已实现，待实机验收）

### 当前状态

库存主文档继续保存在扩展用户配置键 `inventory.v1.document` 下。手动导出仍通过系统保存对话框选择目的地；桌面端首次设置时无需路径输入，插件直接在系统“文档”目录下创建 `JLCEDA-Inventory` 文件夹与固定名称 JSON，试写成功后启用。已有旧版路径继续保留，并可一键迁回默认位置。Web 端明确降级为手动导出。恢复菜单支持选择外部 JSON 或上次恢复前快照，并在覆盖前展示 schema、条目、分类、订单、BOM 流水与批次、项目快照、采购记录和替代料关系数量。

### 目标结构

```text
库存操作
  -> EDA 用户配置（运行时主存储）
    -> 保存成功
      -> 桌面端指定路径备份（可恢复副本）
```

桌面端使用 `SYS_FileSystem.getDocumentsPath()` 获取默认目录、`createDirectoryInFileSystem()` 创建备份文件夹，并通过 `saveFileToFileSystem()` 访问最终绝对路径。这些 API 只在客户端有效、需要扩展外部交互权限并属于 BETA。官方 API 暂不提供“选择任意目录并返回绝对路径”的保存选择器，返回 `void` 的 `saveFile()` 也不能用于记住后续自动覆盖位置，因此当前不提供任意目录选择或路径输入。Web 端必须保留现有手动导出，不得尝试静默访问本地路径。

实机日志确认嘉立创 EDA 专业版 3.2.166 的上述 BETA 适配会把有效文档目录解析为空，并可能因建目录参数名错配返回 `false`。0.5.1 在官方 API 已通过外部交互权限检查后，才使用同一桌面客户端调用链读取实际 `documents` 字段并重试建目录；回退失败时保持自动备份未启用并给出准确错误，不把路径格式问题误报成权限问题。

### P0.1：备份设置与状态（已实现）

- 增加“本地备份设置”，允许启用或关闭自动备份。
- 首次直接创建默认备份文件夹和固定 JSON，并明确展示最终路径；已有旧路径允许继续使用或一键迁回默认位置。
- 路径配置保存在扩展用户配置中，但不得混入库存文档 schema。
- 展示最近一次成功备份时间、文件名和最近一次失败原因。
- 提供“立即测试备份”，在启用前验证权限和可写性。
- Web 端显示“仅桌面客户端支持自动备份”，继续提供手动导出。

### P0.2：自动写入（已实现第一版）

- 添加、编辑、合并、删除、订单导入和状态变化成功后触发备份。
- 固定维护一个最新快照，例如 `jlceda-inventory-latest.json`。
- 第一版只维护用户配置的最新 JSON 快照；带 UTC 时间戳的多版本历史与保留策略尚未实现，避免未经确认地持续占用本地磁盘。
- 备份内容继续使用版本化 `InventoryDocument`，不得引入只存在于备份中的私有库存字段。
- 同一时刻只允许一个备份写入任务，连续变更可以合并为最后一次最新快照。
- 诊断日志只记录步骤、结果和错误类别；完整本地路径默认脱敏，避免泄露用户名和目录结构。

### P0.3：校验恢复（已实现完整替换）

- 支持通过文件选择器读取自动或手动生成的 JSON 备份，也可恢复上次覆盖前保存的恢复点；损坏的内部恢复点不会阻断外部文件恢复。
- 恢复前校验 JSON 结构、schema 版本、条目数量和关键字段。
- 支持读取仍可迁移的旧 schema；不支持的未来 schema 必须拒绝导入。
- 确认窗口展示备份时间、条目数量、当前 revision 和备份 revision。
- 覆盖前先把当前库存导出为恢复前快照；该保护步骤失败时默认中止恢复。
- 恢复成功后写入 EDA 主存储，再刷新已启用的本地最新备份；主恢复失败时还原此前的恢复点。
- 第一阶段只实现“完整替换恢复”；按条目合并应复用订单导入的重复项策略，并作为独立功能评审。

### 验收条件

领域、仓储、控制器和备份协议已完成自动化验证。下列涉及真实路径权限、系统保存对话框和客户端重启的项目仍必须在桌面端验收；Web 端需验证不会调用仅桌面可用的路径 API。

- 桌面端重启后仍能读取备份配置并继续自动备份。
- 每一种库存写操作都能在主存储成功后更新最新快照。
- 路径不存在、权限不足、磁盘写入失败时，库存主操作仍成功且界面明确告警。
- 备份文件损坏、schema 不支持或字段非法时不会覆盖当前库存。
- 恢复前快照、用户确认和恢复后重新备份均可由测试验证。
- Web 端不调用仅桌面可用的路径 API，手动导出保持可用。

## P1：跨设备行为验证

按[跨设备验证清单](cross-device-validation.md)验证 EDA 用户配置的传播、隔离、冲突和容量行为。在验证完成前，不把它描述为官方云同步能力。

如果验证表明配置无法可靠同步，后续同步方案必须通过新的 `InventoryRepository` 适配器实现，不能把同步逻辑写进库存领域规则。独立后端需要另行设计身份认证、加密、离线冲突、删除传播和数据导出能力。

## P2：设计流程联动

- 当前设计只读库存校验、选中器件定位、项目需求快照和显式 BOM 整批出库已实现；所有库存写入都需要用户明确确认。
- 只有官方 API 能可靠确认元器件已经完成放置时，才评估自动扣减库存。
- 自动扣减必须支持撤销、重复放置和失败恢复，不能把“已绑定到鼠标”视为实际使用。
- 商城商品自动补全只考虑获得授权的官方接口，不抓取 HTML，也不在扩展中保存共享密钥。

## 暂不采用

- 不把用户手动编辑的本地 JSON 与 EDA 用户配置做无提示双向同步。
- 不在 Web 端伪造固定本地路径能力。
- 不因本地备份失败而丢弃已经成功写入 EDA 主存储的库存修改。
- 不在未完成双设备验证前宣传跨设备自动同步。
