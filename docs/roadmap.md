# 后续功能路线

本文记录已经明确的可靠性路线及其状态。P0 已在 0.5.0 实现，桌面端指定用例已在 0.5.5 通过；仍需在嘉立创 EDA Web/桌面宿主和多设备环境完成的项目继续保留为专项验证或 P1/P2，实际状态以 README、CHANGELOG、代码和测试记录为准。

## 设计原则

- EDA 用户配置继续作为运行时主存储，保持现有 Web/桌面兼容性和开箱即用体验。
- 本地文件是可恢复的备份副本，不作为可独立修改的第二主数据源。
- 主存储与备份严格按顺序写入：先确认 EDA 主存储成功，再生成本地备份。
- 自动备份失败不得回滚已经成功的库存操作，但必须向用户显示可持续追踪的失败状态。
- 所有恢复操作必须先验证、再预览、后确认，不直接覆盖当前库存。

## P0：指定路径自动备份与恢复（已实现，桌面端指定用例已通过）

### 当前状态

库存主文档继续保存在扩展用户配置键 `inventory.v1.document` 下。“另存库存备份”通过系统保存对话框选择目的地；桌面端首次设置时直接打开原生目录选择器，选定目录后自动创建固定名称 JSON，试写成功才启用。已有旧版路径继续保留，并可随时重新选择目录；取消不会改变设置。Web 端明确降级为“另存库存备份”。恢复菜单支持选择外部 JSON 或上次恢复前快照，并在覆盖前展示 schema、条目、分类、订单、BOM 流水与批次、项目快照、采购记录和替代料关系数量。

### 目标结构

```text
库存操作
  -> EDA 用户配置（运行时主存储）
    -> 保存成功
      -> 桌面端指定路径备份（可恢复副本）
```

桌面端通过 `saveFileToFileSystem()` 写入最终绝对路径。这些文件 API 只在客户端有效、需要扩展外部交互权限并属于 BETA。公开的 `openReadFolderDialog()` 在 3.2.166 中固定返回空数组，也不承诺目录绝对路径，因此 0.5.3 在每次选择前先调用 `SYS_FileSystem.getDocumentsPath()` 完成桌面端校验，再通过受外部交互权限控制的 `SYS_ClientUrl.request()` 请求客户端已有的 `app://api/client/openDir`，打开原生目录选择器。只有用户选中的绝对目录会被保存；取消、权限失败或异常均不修改设置，Web 端也不会调用该内部接口。

0.5.1 实机日志确认默认文档目录回退仍会返回不可用路径，且 `createDirectoryInFileSystem()` 还存在 `folderPath/path` 参数错配。0.5.2 改为目录选择器后，实机日志进一步确认扩展沙箱会禁用全局 `fetch()`，使请求无法到达客户端。0.5.3 改走公开 `SYS_ClientUrl.request()`，并继续直接向用户选择的现有目录写固定 JSON。`app://api/client/openDir` 仍属于当前客户端内部桥接；若官方后续提供可返回目录绝对路径的公开选择器，应优先替换并删除该桥接。

0.5.5 修复停用状态保存后残留已完成队列、导致重新启用后的普通保存或恢复不再刷新的问题。宿主临时拒绝或文件写入异常会在 150 ms 后重试一次，永久失败仍不回滚主库存；正式日志保留脱敏的结果状态、revision、尝试次数和失败分类，恢复日志用相同 revision 与后台备份结果闭环。

2026-07-23 的 `0.5.5` 桌面端指定用例已覆盖原生目录选择、取消、权限拒绝、固定 JSON 创建、停用与重新启用、启用后的连续保存、外部文件恢复和恢复前快照恢复。外部文件恢复与恢复前快照分别在 `revision 52/53` 与自动备份成功记录闭环；重新启用后的两次普通保存连续刷新到 `revision 60/61`。最终固定 JSON 与当前库存导出均为 `revision 61`，SHA-256 同为 `26C4C952CD300CD349C02D8DE2C6A0915872DBB4B4D67DC52C25028B141ED73E`。

### P0.1：备份设置与状态（已实现）

- 增加“本地备份设置”，允许启用或关闭自动备份。
- 首次直接打开原生目录选择器，在选定目录中创建固定 JSON 并明确展示最终路径；已有旧路径允许继续使用或重新选择目录。
- 路径配置保存在扩展用户配置中，但不得混入库存文档 schema。
- 展示最近一次成功备份时间、文件名和最近一次失败原因。
- 提供“立即测试备份”，在启用前验证权限和可写性。
- Web 端显示“仅桌面客户端支持自动备份”，继续提供“另存库存备份”。

### P0.2：自动写入（已实现第一版）

- 添加、编辑、合并、删除、订单导入和状态变化成功后触发备份。
- 固定维护一个最新快照，例如 `jlceda-inventory-latest.json`。
- 第一版只维护用户配置的最新 JSON 快照；带 UTC 时间戳的多版本历史与保留策略尚未实现，避免未经确认地持续占用本地磁盘。
- 备份内容继续使用版本化 `InventoryDocument`，不得引入只存在于备份中的私有库存字段。
- 同一时刻只允许一个备份写入任务，连续变更可以合并为最后一次最新快照。
- 诊断日志只记录步骤、结果和错误类别；完整本地路径默认脱敏，避免泄露用户名和目录结构。

### P0.3：校验恢复（已实现完整替换）

- 支持通过文件选择器读取自动或手动生成的 JSON 备份，也可恢复上次覆盖前保存的恢复点；损坏的内部恢复点不会阻断外部文件恢复。
- 恢复前校验 JSON 结构、schema 版本、条目数量和关键字段。
- 支持读取仍可迁移的旧 schema；不支持的未来 schema 必须拒绝导入。
- 确认窗口展示备份时间、条目数量、当前 revision 和备份 revision。
- 覆盖前先把当前库存导出为恢复前快照；该保护步骤失败时默认中止恢复。
- 恢复成功后写入 EDA 主存储，再刷新已启用的本地最新备份；主恢复失败时还原此前的恢复点。
- 第一阶段只实现“完整替换恢复”；按条目合并应复用订单导入的重复项策略，并作为独立功能评审。

### 验收条件

领域、仓储、控制器和备份协议已完成自动化验证。桌面端指定用例已确认：

- 原生目录选择、取消和外部交互权限拒绝不会留下错误配置。
- 停用期间不写固定文件，重新启用会立即补写当前 revision，后续连续保存持续刷新最新快照。
- 外部 JSON 与恢复前快照均可完整恢复，恢复成功后固定 JSON 与主库存保持一致。

以下项目仍待完整宿主验收，不包含在桌面端指定用例的完成结论中：

- 桌面端重启后仍能读取备份配置并继续自动备份。
- 每一种库存写操作都能在主存储成功后更新最新快照。
- 运行期间路径消失、权限不足或磁盘永久写入失败时，库存主操作仍成功且界面明确告警。
- 备份文件损坏、未来 schema 或字段非法时宿主拒绝覆盖当前库存；恢复写入失败时恢复点仍可用。
- Web 端不调用仅桌面可用的路径 API，“另存库存备份”保持可用。

## P1：跨设备行为验证

按[跨设备验证清单](cross-device-validation.md)验证 EDA 用户配置的传播、隔离、冲突和容量行为。在验证完成前，不把它描述为官方云同步能力。

如果验证表明配置无法可靠同步，后续同步方案必须通过新的 `InventoryRepository` 适配器实现，不能把同步逻辑写进库存领域规则。独立后端需要另行设计身份认证、加密、离线冲突、删除传播和数据导出能力。

## P2：设计流程联动

- 当前设计只读库存校验、选中器件定位、项目需求快照和显式 BOM 整批出库已实现；所有库存写入都需要用户明确确认。
- 只有官方 API 能可靠确认元器件已经完成放置时，才评估自动扣减库存。
- 自动扣减必须支持撤销、重复放置和失败恢复，不能把“已绑定到鼠标”视为实际使用。
- 商城商品自动补全只考虑获得授权的官方接口，不抓取 HTML，也不在扩展中保存共享密钥。

## 暂不采用

- 不把用户手动编辑的本地 JSON 与 EDA 用户配置做无提示双向同步。
- 不在 Web 端伪造固定本地路径能力。
- 不因本地备份失败而丢弃已经成功写入 EDA 主存储的库存修改。
- 不在未完成双设备验证前宣传跨设备自动同步。
