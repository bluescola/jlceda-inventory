# 元器件统一添加表单与窗口层级边界（已实现）

> 状态：已在 `0.4.0` 实现，自动化测试已通过；仍需在嘉立创 EDA V3 Web 端和桌面端完成输入、焦点、滚动、商城跳转和窗口尺寸实机验证。

## 当前问题

当前两个添加入口使用不同流程：

- “按立创 C 编号添加”先单独输入 C 编号，再查询 EDA 或确认商城信息，最后依次填写数量、数量类型和存放位置。
- “添加自定义元器件”依次填写名称、制造商、厂家型号、供应商编号、数量、数量类型和存放位置。

这些步骤把同一条库存记录拆成多个连续弹窗。用户不能一次检查完整信息，取消或返回时也容易丢失已经填写的内容。

## 目标

两个入口统一使用同一套一次性多字段表单和同一套保存流程：

1. 用户在一个窗口中看到并填写完整元器件身份、库存、分类、位置和备注。
2. “按立创 C 编号添加”和“添加自定义元器件”只保留入口模式差异，不维护两套字段、校验或库存输入流程。
3. EDA 模型查询、商城确认、重复项比对和错误提示使用大表单上方的小型子窗口或状态区域，大表单不关闭，草稿和滚动位置不丢失。
4. 用户只在最终点击“保存”后写入库存；查询、自动填充、子窗口确认和取消均只修改当前草稿。

## 统一表单

建议新增 `InventoryCreatePanel` 端口，两个菜单入口以不同模式打开同一个表单：

```ts
type InventoryCreateMode = 'custom' | 'lcsc';

interface InventoryCreateRequest {
	mode: InventoryCreateMode;
	initial?: Partial<InventoryCreateDraft>;
}
```

目标布局：

```text
+--------------------------------------------------------------------+
| 添加元器件                                                         |
|--------------------------------------------------------------------|
| 立创 C 编号          [ C233991              ] [查询 EDA 模型]      |
| EDA 模型状态         [未查询 / 查询中 / 可用 / 无模型 / 失败]      |
| 供应商编号           [                      ]                       |
| 名称 *               [                      ]                       |
| 制造商               [                      ]                       |
| 厂家型号             [                      ]                       |
| 封装                 [                      ]                       |
| 描述                 [                                      ]      |
| 数量状态             [准确] [大概] [未知] [已用完]                 |
| 数量                 [                      ]                       |
| 一级/二级分类        [一级分类] [二级分类]                         |
| 存放位置             [                      ] [选择已有位置]        |
| 备注                 [                                      ]      |
|--------------------------------------------------------------------|
|                                              [取消] [保存到库存]    |

```

字段边界：

| 字段 | 按 C 编号添加 | 自定义添加 |
| --- | --- | --- |
| 立创 C 编号 | 必填，格式为 `C` 加数字 | 可选；填写有效编号后可主动查询 EDA 模型 |
| 供应商编号 | 默认使用 C 编号，可修改 | 可选 |
| 名称 | 必填，可由 EDA 模型建议填充 | 必填 |
| 制造商、厂家型号、封装、描述 | 可由 EDA 模型或用户确认的商城信息建议填充 | 可选，用户直接填写 |
| 数量状态与数量 | 使用统一库存规则 | 使用统一库存规则 |
| 分类、存放位置、备注 | 可选，使用相同控件 | 可选，使用相同控件 |

两个模式必须复用同一个表单状态、验证器、提交数据结构和组件；不得只把两个旧流程放进外观相似的不同窗口。

## 数量与库存规则

- 数量状态使用单选分段控件：准确、大概、未知、已用完。
- 选择“准确”或“大概”时数量必填，必须是非负安全整数；负数和小数阻止保存并在数量字段旁提示。
- 数量为 `0` 时自动切换为“已用完”；如果用户随后取消“已用完”，恢复切换前显示的数量和准确/大概状态。
- 选择“未知”时不要求数量，保存为 `quantity: null`、`precision: unknown`，不得把空白伪装为 `0`。
- 选择“已用完”时保存数量 `0` 和准确状态，但只有最终保存后才修改库存。
- 存放位置继续支持自由输入和“选择已有位置”；从已有位置选择只回填草稿。
- 分类默认为“未分类”。选定一级分类后才显示或启用对应二级分类；保存时只写最终选中的实际分类 ID。

## 按立创 C 编号添加

### 打开与查询

- 打开统一表单后直接聚焦 C 编号字段，不再先弹出单字段 C 编号输入框。
- 输入合法 C 编号后可以按 Enter 或点击“查询 EDA 模型”；查询在当前表单会话中执行，表单保持打开。
- 查询期间只禁用 C 编号和查询按钮，其他已填写字段、数量、分类、位置和备注保持可见，不清空草稿。
- C 编号变化后，旧查询结果立即标记为过期；旧请求晚到的响应不得覆盖新编号状态。

### EDA 模型命中

- 命中时展示模型摘要和可采用字段，不直接覆盖用户已经修改的值。
- 空字段可以自动填充；非空字段与模型值不同时显示逐字段“采用建议”，由用户决定。
- EDA 模型引用只在最终保存时随库存记录写入。

### EDA 模型未命中或查询失败

- 可以打开立创商城外部搜索，但统一添加表单保持在原位置，已填写字段不丢失。
- 商城是否存在该商品、商城字段确认和“仍作为自定义记录保存”的选择都回到同一表单完成，不再追加一串原生输入框。
- 商城商品信息与 EDA 模型状态继续独立：商城有商品但系统库无模型时仍允许保存库存。

## 添加自定义元器件

- 使用与按 C 编号添加完全相同的表单，初始标题和说明表明当前是“自定义”模式。
- 名称是唯一必填身份字段；C 编号、供应商编号、制造商、厂家型号、封装和描述均可选。
- 默认不自动打开商城，也不自动查询 EDA 模型。
- 用户填写有效 C 编号后，可以在原表单中主动点击“查询 EDA 模型”；命中后使用与 C 编号模式相同的建议和确认逻辑。
- 如果填写 C 编号并成功关联模型，记录仍按最终证据确定来源，不能仅因为从“自定义添加”入口进入就丢弃模型引用。

## 大窗口与小窗口通用规则

统一添加表单属于当前任务的大窗口。模型结果确认、重复项比对、删除或覆盖确认、字段冲突提示属于其上方的小窗口或模态层：

- 打开小窗口时，大表单保持挂载并保留字段值、滚动位置、当前焦点上下文和已经采用的建议。
- 小窗口始终位于大表单上方；点击大表单背景不能把小窗口压到下层，只能保持遮罩或把活动小窗口带到前方。
- 小窗口取消后回到触发它的按钮或字段，不重建大表单。
- 小窗口确认后只更新相关草稿或状态；除最终“保存到库存”外，不得提前写库存主文档。
- 同一时刻只允许一个需要用户决策的小窗口。新的确认请求必须等待当前小窗口结束，避免窗口互相覆盖。
- 这套规则作为通用交互模式复用于库存总览、完整编辑、统一添加、模型匹配、重复项比对以及后续设置页面。

如果统一添加从未来的库存总览按钮打开，则库存总览是父窗口、统一添加表单是第一层子窗口、模型确认或重复项比对是第二层子窗口；关闭内层时逐级返回，库存总览始终不关闭。

## 重复记录与保存

- 点击“保存到库存”后才执行完整验证和重复身份检查。
- 如果发现重复记录，在统一添加表单上方打开比对窗口，显示现有记录与待添加草稿；大表单不关闭。
- 取消合并返回原草稿，用户可以继续修改；确认合并后才执行 revision 校验和原子合并。
- 保存成功后显示结果摘要。由顶级菜单打开时可以关闭添加表单；由库存总览打开时关闭添加子窗口并局部刷新总览。
- 保存失败或 revision 冲突时保留完整草稿，显示错误并允许重试，不得让用户重新填写全部字段。

## 兼容与降级

- 正常路径不得调用多个连续的 `showInputDialog` 或 `showSelectDialog` 收集表单字段。
- IFrame 初始化失败时不得静默进入长串分步输入。`0.4.0` 会显示明确错误并记录诊断，不提供行为不同的自动回退。
- 未来如果增加兼容分步录入，必须由用户在错误后主动选择，并且只作为宿主 API 不可用时的应急路径，不能作为验收统一表单功能的替代证明。
- Web 和桌面端必须分别验证输入、滚动、焦点、模型查询、商城跳转、重复比对、取消恢复和窗口层级。

## 协议与模块边界

- `InventoryCreatePanel` 负责创建草稿和用户交互；`InventoryService` 继续负责验证分类引用、重复判断、合并和持久化。
- 统一创建表单与完整编辑表单共享字段定义、数量状态控制、位置选择、分类选择、文本归一化和校验模块，但创建与更新使用独立请求类型，不能混淆 revision 语义。
- 现有 `ProductDetailsForm` 的商品身份字段能力可以下沉为统一表单的可复用区块，不再由控制器把它与多个库存原生弹窗串联。
- 查询 EDA 模型和打开商城仍由平台适配层执行，表单只发送带操作 ID 的明确请求并接收阶段响应。
- 诊断记录模式、步骤、耗时、字段名、长度和结果状态，不记录完整 C 编号、元器件名称、备注、位置或表单草稿。

## 明确不在本次范围

- 从商城网页自动抓取商品字段。
- 自动猜测用户分类或存放位置。
- 在用户未确认时覆盖已填写字段。
- 因输入 C 编号而强制要求商城和 EDA 模型同时存在。
- 把表单草稿写入库存主文档作为自动保存。
- 同时打开多个需要用户决策的小窗口。

## 验收条件

- 两个菜单入口打开同一套多字段表单，字段顺序、数量规则、分类和位置控件一致。
- 按 C 编号添加不再出现独立 C 编号、数量、精度和位置输入弹窗；自定义添加不再逐个询问名称、制造商、型号和库存信息。
- EDA 查询命中、未命中、失败、取消和过期响应都不清空表单草稿。
- 自定义模式填写 C 编号后可以在同一表单主动查询和关联 EDA 模型。
- 重复项比对位于大表单上方，取消后完整草稿仍在，确认后才写入或合并。
- 所有小窗口保持在大表单上方，关闭后恢复大表单的位置、内容和焦点；从总览进入时还必须保留总览状态。
- 只有最终保存会写库存，取消任意中间步骤不产生半条记录。
- 简体中文和英文字段、状态、错误、按钮和辅助文本同步。
- Web 与桌面端完成统一表单及最多三层窗口关系的实机测试。

## 已确认决策

1. 两个添加入口复用同一个 `InventoryCreatePanel`，不再维护两套录入流程。
2. C 编号模式要求 C 编号，自定义模式只要求名称；其余字段和保存规则一致。
3. 自定义模式允许填写 C 编号并主动查询 EDA 模型。
4. 数量状态统一为准确、大概、未知和已用完，并保留已用完取消后的数量恢复规则。
5. 分类、存放位置和备注都在首次添加时一次填写，不在保存前后继续追加弹窗。
6. EDA 查询、商城确认和重复项比对作为大表单上方的小窗口或状态流程，不关闭或重建大表单。
7. 正常流程禁止连续原生分步录入；当前不提供静默回退，未来兼容分步录入也必须由用户在表单启动失败后主动选择。
