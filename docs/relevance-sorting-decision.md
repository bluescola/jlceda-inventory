# 库存总览“相关性”排序待决策记录

> 状态：待确认。计划在下一轮讨论中确定产品语义后再修改代码、文案和测试。本文件记录问题和决策框架，不代表已经选定最终方案。

## 背景

库存总览同时承担两种行为：

1. 空搜索时浏览全部、未分类或某个用户分类；
2. 输入 C 编号、型号、名称、制造商或封装后查找元器件。

排序下拉框目前在两种行为中都显示“相关性”，但“相关性”只有在存在查询词时才有自然含义。空搜索时必须选择一个回退顺序；如果回退顺序隐含分类分组，用户从二级分类切回“全部”后会看到原二级分类的元器件仍完整排列在最前面，容易误判为列表没有刷新。

本问题只涉及结果展示顺序，不涉及库存写入、分类变更或搜索范围。

## 当前临时行为（0.4.12）

### 有查询词

候选记录先经过分类范围、库存状态和 EDA 模型状态过滤，再由共享搜索逻辑排序：

1. 完全匹配；
2. 前缀匹配；
3. 包含匹配；
4. 同一匹配等级内，有库存优先；
5. 仍相同时，更新时间较新的优先。

参与匹配的字段为 C 编号、供应商编号、厂家型号、名称、制造商和库存封装。当前这些字段没有单独的权重。

### 无查询词

“相关性”临时回退为：

1. 有库存优先于已用完；
2. 全局名称自然升序；
3. ID 作为稳定兜底。

当前回退不按分类分组。只有用户显式选择“分类顺序”时，才按用户维护的一级、二级分类顺序分组，再按名称排序。

## 已确认的事实

- 从二级分类切换到“全部”必须重新计算完整结果，不能复用子分类结果数组或其导航顺序。
- 排序结果必须只由当前查询、分类、筛选、排序选项和库存快照决定，不能受之前浏览过哪个分类影响。
- 显式“分类顺序”可以按用户分类分组；“相关性”是否包含分类权重尚未确认。
- 用户已经确认当前问题不是简单的下拉框事件或页面未刷新，而是需要重新定义“相关性”的产品含义。
- 相关性排序不得自动修改分类、库存或模型状态。

## 下一轮需要确定的问题

| 决策项 | 需要确认的选择 |
| --- | --- |
| 空搜索是否保留“相关性” | 保留并定义回退顺序；或空搜索时改称“默认顺序”；或自动切换到另一个明确排序 |
| 空搜索第一优先级 | 有库存、名称、最近更新、分类、最近使用，或其他业务维度 |
| 分类是否参与相关性 | 完全不参与；仅在当前分类内作为兜底；或在“全部”中也参与 |
| 搜索字段是否加权 | C 编号、厂家型号、名称、制造商、封装是否应有不同权重 |
| 文本匹配与库存状态顺序 | 先保证完全匹配，再区分库存；还是有库存记录可以超过更高文本匹配等级 |
| 已用完记录的位置 | 始终置后；仅同匹配等级置后；或完全由显式库存筛选控制 |
| 同分兜底 | 名称、更新时间、C 编号、创建时间或稳定 ID 的顺序 |
| 排序名称 | “相关性”是否足够准确，是否需要“最佳匹配”“默认顺序”等更明确文案 |

## 候选方向（未选择）

### 方向 A：相关性仅服务搜索

- 有查询词时使用文本匹配等级和业务权重。
- 无查询词时使用一个明确、稳定的默认顺序，例如“有库存优先 + 名称”。
- UI 可以在空搜索时把文案显示为“默认顺序”，避免声称存在搜索相关性。

优点是语义清楚、导航历史不会影响顺序；需要额外确认默认顺序是否符合日常盘点习惯。

### 方向 B：相关性代表操作优先级

- 无论是否搜索，都把有库存、模型可用、最近更新等维度组合为操作优先级。
- 文本匹配存在时仍需要决定文本分数和操作优先级谁先比较。

优点是可能更贴近日常使用；缺点是排序原因不直观，而且当前没有“最近使用”等可靠数据。

### 方向 C：按浏览上下文定义默认顺序

- 当前分类内可以按名称或库存排序；“全部”使用另一套全局顺序。
- 显式搜索仍使用统一的搜索相关性。

优点是能针对浏览场景优化；缺点是同一个排序名称在不同分类下可能表现不同，必须通过文案或控件状态避免歧义。

## 明天的验收样例

决策时至少对以下固定样例逐条给出期望顺序：

1. 二级分类中有 `Alpha`、`Charlie`，其他分类有 `Bravo`、`Delta`；从二级分类切回“全部”后，相关性应如何排列。
2. 查询 `C100` 时，同时存在精确命中的有库存/已用完记录、`C1000` 前缀记录和名称中包含 `C100` 的记录。
3. 查询 `0603` 时，命中库存封装、EDA 模型封装、名称和厂家型号的记录应如何比较。注意当前搜索只匹配库存封装，是否加入 EDA 模型封装也需明确。
4. 同一个查询分别精确命中 C 编号和名称时，字段是否等权。
5. 空搜索且全部记录都有库存时，名称、分类、更新时间三种维度谁优先。
6. 同分记录在刷新、切换分类、分页和重新打开总览后必须保持稳定顺序。

建议用导出的库存备份做只读回放，再用上述合成数据覆盖备份中没有的冲突组合。不得用真实库存写入来验证排序。
